🔍 Vulnerability Findings
CWE-352 Missing CSRF protection on login endpoint MEDIUM ❌ UNMITIGATED

The login endpoint accepts POST requests without validating anti-CSRF tokens. An attacker can trick a victim's browser into submitting a login request with the attacker's credentials, causing the victim to be logged into the attacker's account (login CSRF). If the application uses the JWT token for authorization, this could lead to privilege escalation or account takeover.

Fix: Require CSRF tokens on all state-changing requests. Use SameSite cookies.
CWE-269 Hardcoded JWT Secret Key Leading to Privilege Escalation HIGH ❌ UNMITIGATED

The application uses a hardcoded secret key ('your-super-secret-key-change-in-production') to sign JWT tokens. An attacker who extracts this key (e.g., via source code leak) can forge valid tokens for any user, bypassing authentication and escalating privileges to access any account, including those with PHI access.

Fix: Review the CWE definition and apply defense-in-depth controls.
CWE-770 Unbounded memory growth in rate limiting storage MEDIUM ⚠️ PARTIAL

The record_failed_attempt function appends timestamps to a per-IP list without ever removing entries for IPs that cease login attempts. Over time, the dict accumulates entries for every unique IP that ever attempted a failed login, leading to unbounded memory consumption and potential denial of service.

Fix: Review the CWE definition and apply defense-in-depth controls.
CWE-532 Sensitive Data (Username) Written to Application Logs MEDIUM ⚠️ PARTIAL

The login endpoint logs the username in both successful and failed login attempts via app.logger.warning/info. This exposes potentially identifying information (username) in log files, which could be accessed by unauthorized personnel, violating HIPAA minimum necessary and privacy requirements.

Fix: Review the CWE definition and apply defense-in-depth controls.
